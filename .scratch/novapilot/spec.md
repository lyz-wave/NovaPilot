# NovaPilot 3.0：科研客户技术支持与咨询服务体系

Status: ready-for-agent

Scope: 完整产品蓝图 + 12 周可信 MVP

Primary test seam: 一次完整科研咨询旅程

## Problem Statement

科研客户在实验设计、样本处理、测序平台选择、数据分析和结果解释过程中，需要反复补充样本条件、研究目标、预算、周期与合规约束，并判断建议是否适用于当前项目。现有 AI 科研助手虽然具备文献检索、方案生成、生信分析和写作能力，但仍以单轮聊天为中心：项目事实容易在多轮咨询中漂移，关键建议与证据没有形成可审计关联，遇到证据冲突或高风险条件时缺少可靠的强制转接，专家接管后还要重新询问上下文，咨询结果也无法稳定沉淀为后续服务资产。

客户真正需要的不是“AI 回答了多少问题”，而是在证据充分时获得可执行、可比较、可审批的科研方案，在证据不足或风险较高时被准确追问并顺畅交给正确的专家。诺禾致源同时需要在不牺牲科学可信度、数据安全和客户控制权的前提下，降低重复咨询与人工整理成本，让服务适配说明和销售动作自然衔接，并把专家修改、失败咨询和反馈转化为受治理的候选知识或候选策略。

可信 MVP 必须首先解决 FFPE/RNA 场景下的实验方案设计和平台选型，服务博士后、研究生等操作用户以及负责审批的 PI。系统的北极星指标是安全自助解决率，而不是单纯自助率；引用有效率、高风险转接召回率和自信错答表现是不可被业务增长目标覆盖的安全发布门禁。

## Solution

NovaPilot 以科研项目状态为主线，以版本化科研决策卡为正式交付物，以聊天作为收集条件和解释决策的交互界面。系统通过最小必要追问确认项目事实，使用统一科研实体支持中、英、日三语跨语言检索，从当前有效的内部 SOP、公开 SCI 文献、FAQ 和脱敏案例中检索证据，并把每条关键建议与来源、版本、适用样本、有效期和证据状态进行结论级绑定。

系统按照样本、合规、证据充分性、方案可逆性和错误后果进行风险分级。低风险且证据门禁全部通过时，可发布正式决策卡；中风险发布带明确限制的暂行决策卡并允许申请复核；高风险、证据冲突、证据不足、非标或极低质量样本、人源合规风险以及客户主动要求人工时，触发强制转接，停止输出可直接执行的最终方案，并通过一次性交接包把项目事实、证据、AI 尝试、风险和待决策项同步给解决方案专家。

NovaPilot 只主责科研项目状态、科研决策卡版本和证据链。CRM 继续主责客户、授权和商机；工单系统主责专家分派和 SLA；知识平台主责正式 SOP。科学建议、服务适配说明和授权式转化分层呈现，只有客户主动获取报价、预约专家或明确同意联系后，才创建或更新 CRM 商机。

服务过程中的专家修改、失败咨询、检索缺口和质量事件只生成候选知识或候选策略。候选项必须经过知识 Owner 审核、NovaBench 回归、人工批准和门禁式灰度才能成为正式知识；系统不得自动修改模型权重、智能体代码、生产工作流或正式 SOP。

## User Stories

1. As an 操作用户, I want to 从现有 AI 科研助手、官网组件或外部入口进入同一咨询服务, so that 我无需判断应该使用哪个独立工具。
2. As an 匿名操作用户, I want to 在登录前进行轻量咨询, so that 我可以先判断 NovaPilot 是否适合当前科研问题。
3. As an 操作用户, I want to 在需要保存项目、上传材料或申请专家时完成登录, so that 我的科研项目状态能够被安全保存和复用。
4. As an 操作用户, I want to 选择或自动识别当前使用的中文、英文或日文, so that 我可以使用最熟悉的语言表达科研问题。
5. As an 操作用户, I want to 在同一项目中切换中、英、日语言, so that 切换语言后已确认的项目事实不会丢失或被重新猜测。
6. As an 操作用户, I want to 录入研究目标、物种、样本类型、数量、分组、质量、预算和周期, so that 系统能够建立当前项目的约束条件。
7. As an 操作用户, I want to 从文本或附件中提取候选项目事实, so that 我不必重复手工填写已有材料。
8. As an 操作用户, I want to 在候选项目事实写入正式项目状态前逐项确认, so that 模型推断不会被误当成客户事实。
9. As an 操作用户, I want to 看见每个项目事实的来源、确认状态和版本, so that 我知道哪些信息来自自己、AI 或专家。
10. As an 操作用户, I want to 修正已确认但后来发现有误的项目事实, so that 后续方案能够基于最新条件重新评估。
11. As an 操作用户, I want to 让系统只提出会改变方案的最小必要问题, so that 我不会被冗长问卷阻断咨询。
12. As an 操作用户, I want to 看见某个追问为什么必要, so that 我理解该信息对平台选择、风险或预算的影响。
13. As an 操作用户, I want to 在暂时不知道答案时获得可行的检测方法或条件性路径, so that 咨询仍可推进但不会产生虚假确定性。
14. As an 操作用户, I want to 比较 FFPE/RNA 实验的主方案与备选方案, so that 我可以基于样本质量、研究目标和限制做选择。
15. As an 操作用户, I want to 比较适用测序平台的通量、准确性、样本量、周期、预算区间和下游兼容性, so that 平台选择透明且可解释。
16. As an 操作用户, I want to 看到方案适用边界、失败风险、替代路径和待验证步骤, so that 我不会把一般建议误用到非标准样本。
17. As an 操作用户, I want to 区分科学结论、诺禾服务适配说明和销售动作, so that 产品推荐不会污染科研判断。
18. As an 操作用户, I want to 获得版本化科研决策卡而不是只保留聊天记录, so that 咨询结果可导出、复核和继续推进。
19. As an 操作用户, I want to 在科研决策卡中看到已确认条件、主备方案、平台参数、风险、证据、预算与周期区间, so that 我能完整理解建议。
20. As an 操作用户, I want to 看到科研决策卡的草稿、暂行、专家待审、正式和已批准状态, so that 我不会误把未审核建议当成最终方案。
21. As an 操作用户, I want to 比较科研决策卡不同版本之间的事实、证据与建议变化, so that 我可以理解方案为何改变。
22. As an 操作用户, I want to 导出适合 PI 阅读的决策摘要, so that 我可以在实验室内部快速完成讨论。
23. As a 决策用户, I want to 查看主方案、备选方案、证据强度、成本、周期与风险摘要, so that 我可以作出项目审批决定。
24. As a 决策用户, I want to 批准、退回补充信息或要求专家复核科研决策卡, so that 决策责任和下一步动作明确。
25. As a 决策用户, I want to 查看建议与当前有效 SOP 或文献的直接关系, so that 我能够判断方案的科学依据。
26. As an 操作用户, I want to 点击关键建议查看对应证据, so that 我可以核实证据是否真正支持该结论。
27. As an 操作用户, I want to 看到 DOI 或 PMID 的校验状态以及原文标题, so that 我不会依赖伪造或失效的引用。
28. As an 操作用户, I want to 在内部 SOP 与外部文献冲突时看到冲突说明, so that 系统不会通过隐藏证据制造确定性。
29. As an 操作用户, I want to 在证据不足时收到明确的条件缺口和下一步建议, so that 我知道补充什么后才能继续决策。
30. As an 操作用户, I want to 在高风险场景中让 AI 停止给出可直接执行的最终方案, so that 错误建议不会被带入实验。
31. As an 操作用户, I want to 主动申请人工专家, so that 我可以在任何阶段取得人工支持。
32. As an 操作用户, I want to 在转接时看到原因、预计认领时间和处理状态, so that 我知道问题没有消失在系统中。
33. As a 解决方案专家, I want to 收到一次性交接包, so that 我无需重新询问客户目标、项目事实和 AI 已尝试内容。
34. As a 解决方案专家, I want to 看见证据冲突、失败项、风险等级和待决策问题, so that 我可以直接开展实质性判断。
35. As a 解决方案专家, I want to 在一个工作台中认领、修改、批准、拒绝或退回科研决策卡, so that 专家协作和责任可追踪。
36. As a 解决方案专家, I want to 看到 AI 与专家版本之间的差异, so that 修改内容可以被审计并形成候选知识。
37. As an 操作用户, I want to 在专家处理后继续原项目和原语言的咨询, so that 人机协同不会造成上下文断裂。
38. As a 服务运营人员, I want to 监控专家认领与首次实质性回复 SLA, so that 超时任务可以升级和补偿。
39. As a 服务运营人员, I want to 查看安全自助解决率、强制转接、重复开单和满意度, so that 我能识别服务瓶颈而非只看聊天量。
40. As a 服务运营人员, I want to 把低分、客户纠错、引用失效和重复开单转成质量事件, so that 问题有责任人、原因和关闭记录。
41. As an 操作用户, I want to 对每次回答标记有用、无用及原因, so that 系统能够捕捉即时体验问题。
42. As an 操作用户, I want to 在科研决策卡交付后评价整体满意度, so that 完整服务结果可以被衡量。
43. As an 操作用户, I want to 在七日后反馈方案是否采用及遇到的问题, so that 建议的实际有效性能够被验证。
44. As a 知识 Owner, I want to 审核从专家修改和失败咨询中提炼的候选知识, so that 未验证经验不会直接进入生产回答。
45. As a 知识 Owner, I want to 为候选知识设置证据、作用域、反例、版本和有效期, so that 正式知识的适用边界可治理。
46. As a 产品负责人, I want to 让候选策略经过 NovaBench、审批和门禁式灰度, so that 自进化不会成为不可审计的生产改写。
47. As a 产品负责人, I want to 在安全门禁退化时停止放量或回滚, so that 业务指标不能掩盖科学安全问题。
48. As a 合规负责人, I want to 阻止敏感科研数据进入外部模型, so that 内部 SOP、客户文件和人源信息留在受控环境。
49. As a 合规负责人, I want to 审计模型、检索、工具、决策卡、工单、CRM 和反馈事件, so that 数据访问和业务动作可追溯。
50. As an 操作用户, I want to 查看、撤回授权并发起数据删除请求, so that 我能够控制自己的资料与联系许可。
51. As an 操作用户, I want to 在主动获取报价或预约专家时明确授权联系, so that 普通技术咨询不会自动转成销售线索。
52. As a 销售运营人员, I want to 只接收带授权事件和咨询摘要的商机, so that 跟进合规且不需要重新理解客户需求。
53. As a 销售运营人员, I want to 看见服务适配理由、限制和替代选择, so that 销售沟通不会夸大科学适配性。
54. As a 产品运营人员, I want to 聚合高频问题、检索缺口和潜在服务需求, so that 咨询数据能够反哺产品推广和知识建设。
55. As an 企业研发用户, I want to 在完整产品阶段获得组织权限、标准化流程、SLA 与知识产权控制, so that 企业项目可以规模化使用服务。
56. As a 研究人员, I want to 在完整产品阶段获得数据分析方法推荐, so that 分析流程与样本、设计和研究目标一致。
57. As a 研究人员, I want to 在完整产品阶段获得分层的结果解释和论文写作支持, so that 观察事实、可能机制、证据强度和替代解释不会混淆。
58. As a 平台运维人员, I want to 查看 trace、延迟、Token、检索、工具、转接和成本指标, so that 故障与性能退化可以快速定位。
59. As a 系统管理员, I want to 通过角色与属性控制客户、PI、专家、Owner、运营、销售和管理员权限, so that 每个角色只能访问完成任务所需的数据。
60. As a 评测人员, I want to 在历史影子、专家内测、邀请客户和不超过 10% 灰度阶段使用同一套金标门禁, so that 上线结论可比较且可复现。

## Implementation Decisions

- 产品采用完整产品蓝图与可信 MVP 双层范围。可信 MVP 在 12 周内聚焦 FFPE/RNA 实验方案设计与平台选型，完整产品再扩展至四大场景、企业体验、生信执行和更高阶运营能力。
- 可信 MVP 的主要操作用户是博士后与研究生，PI 是决策用户。企业研发用户属于完整产品范围，不作为首期指标与流程优化的主要样本。
- 科研决策卡是核心聚合根和正式交付物；聊天消息不是权威业务记录。科研项目状态包含项目事实、决策卡版本、证据关系、待办和专家状态。
- 科研决策卡状态机为：草稿 → 待补充条件 → 证据审查 → 风险判定 → 暂行／专家待审／正式 → PI 已批准 → 已交付 → 已关闭。发布新版本后保留旧版并记录替代关系。
- 科研决策卡必须保存身份与版本、研究目标、项目事实、主方案、备选方案、关键参数、证据、适用边界、风险、服务适配说明、预算与周期区间、协作状态和审计信息。
- 项目事实必须包含来源、提取时间、置信度、客户确认状态、版本和可见范围。AI 提取内容在客户确认前不得作为正式项目事实影响最终方案。
- 最小必要追问模块根据缺失事实对方案分支、风险和证据适用性的影响排序；每轮最多提出三个关键问题，并解释追问原因。
- 三语服务共享一套科研知识和统一科研实体，不维护三套独立翻译知识库。跨语言检索将客户语言映射至统一实体，扩展查询后检索原始语言资料，并以客户当前语言生成结果；引用保留原始标题、术语和标准缩写。
- 循证 RAG 的数据源分层为：当前有效内部 SOP、公开文献、FAQ、脱敏历史客服和产品规则。操作性要求优先引用当前有效 SOP，方法学判断使用 DOI/PMID 已校验文献。
- 检索采用关键词、向量和科研实体扩展的混合召回，并使用重排器排序。检索与生成必须遵守租户、角色、地区、产品线和文档密级权限。
- 每条关键建议必须建立结论级证据绑定，记录证据来源、版本、适用样本、作用域、有效期和证据强度。证据缺失时不能形成正式决策卡。
- 内部 SOP 与外部文献发生实质冲突时，系统显示冲突内容并触发强制转接，不得静默选择有利来源。
- 科研 Reviewer 独立检查结论—证据覆盖、引用有效性、科研实体与数值单位、适用边界、风险等级和销售措辞。Reviewer 与生成器不得共享隐藏的结论状态。
- 风险等级由样本、合规、证据充分性、方案可逆性和错误后果共同决定，不以模型置信度单独判定。
- 低风险且全部证据门禁通过时可由 AI 发布正式决策卡；中风险只能发布显式受限的暂行决策卡；高风险必须由专家批准后才能形成正式决策卡。
- 证据不足或冲突、非标或极低质量样本、人源合规风险、客户主动要求人工时必须强制转接。强制转接后 AI 只展示已确认事实、转接原因和待决策项，不输出“立即执行”“最终推荐”等可执行结论。
- 专家转接通过一次性交接包进行，至少包含客户目标与当前语言、已确认和冲突的项目事实、已核验证据、AI 尝试、Reviewer 结果、风险等级、转接原因和待决策项。
- 专家工作台支持技能路由、队列、认领、退回补充、修改、批准、拒绝、重新分派和关闭。自动分派或通知不计为专家认领。
- 试点 SLA 目标为专家在工作时间 30 分钟内认领、4 个工作小时内给出首次实质性回复；系统记录超时、升级和补偿。
- 科学结论、服务适配说明和销售动作使用独立的数据对象与状态。服务适配说明必须展示匹配理由、限制和可替代选择。
- CRM 写入必须由授权事件触发。允许的授权事件包括主动获取报价、预约专家或明确同意联系；普通咨询、资料上传和未授权服务回访不得创建商机。
- NovaPilot 是科研项目状态、项目事实、决策卡版本与证据链的主系统；CRM 是客户、授权和商机的主系统；工单系统是专家分派与 SLA 的主系统；知识平台是正式 SOP 的主系统。
- 系统集成以幂等 API 和领域事件同步标识与状态，不复制另一套 CRM、工单或正式知识主数据。失败写入进入补偿队列并提供人工修复入口。
- 核心实体包括 User／Organization、ResearchProject、ProjectFact、Consultation、DecisionCard、Recommendation、Evidence、RiskAssessment、ExpertCase、CandidateKnowledge、ConsentEvent、QualityEvent 和 EvaluationCase。
- 对外接口围绕项目、事实确认、咨询、证据、决策卡审批、专家转接、授权和反馈设计；所有写操作必须支持幂等键、租户隔离、版本检查和 trace ID。
- 核心事件至少包括 ProjectFactConfirmed、DecisionCardSubmittedForReview、MandatoryEscalationTriggered、ExpertCaseAccepted、DecisionCardApproved、ConsentGranted、ConsentWithdrawn、QualityEventOpened、CandidateKnowledgeCreated、KnowledgeVersionActivated 和 KnowledgeVersionRolledBack。
- 所有事件记录 event ID、trace ID、租户、操作者、时间、来源版本和幂等键，并可关联模型、提示、检索、知识和规则版本。
- 自进化仅能自动更新客户已确认的当前项目事实，或生成候选知识与候选策略。候选项不能直接参与全局生产回答。
- 候选知识必须包含触发来源、建议动作、证据、作用域、反例、Owner、版本、有效期和风险。进入正式知识前必须经过 Owner 审核、NovaBench 回归、人工批准和门禁式灰度。
- 模型权重、智能体代码、生产工作流和正式 SOP 不得由系统自动改写上线；任何版本激活必须可审计、可灰度、可停止和可回滚。
- 敏感科研数据必须留在受控私有环境。外部模型仅可在政策允许时处理脱敏问题和公开文献，且供应商不得将输入用于训练。
- 无法可靠脱敏或涉及人类遗传资源合规时，只能使用私有模型并按规则触发人工审核。提示、检索片段、日志和中间结果同样受数据出域策略约束。
- 身份采用统一认证，授权采用 RBAC 与 ABAC 组合。客户仅访问被授权项目；PI 可审批但不能改写事实；专家仅访问认领或被授权项目；知识 Owner 不得查看无关客户原始数据；销售仅访问已授权商机摘要。
- 文件上传执行恶意文件扫描、类型白名单、大小限制、内容隔离和租户级权限。数据访问、撤回授权、导出和删除请求必须有审计轨迹。
- 实时体验闭环包含回答级“有用／无用＋原因”、决策卡完成后 CSAT、交付七日后的方案有效性回访，以及自动生成的质量事件。
- 质量事件由低分、客户纠错、引用失效、重复开单或重大风险触发，必须包含事件类型、影响范围、临时处置、责任人、长期动作和关闭证据。
- 北极星指标为安全自助解决率。安全发布门禁包括引用有效率不低于 98%、高风险转接召回率不低于 95%、自信错答和重大投诉不相对当前灰度阶段恶化、P0 阻断级缺陷为零、敏感数据越权或出域事件为零。
- 业务指标作为建立基线后的试点目标，而非硬发布条件：安全自助解决率目标 55%–65%，首次响应时间降低至少 90%，人工技术支持效率提升 35%–45%，CSAT 提升 8–12 个百分点，有效新客户触达增长 30%–50%。
- 发布采用门禁式灰度：历史影子评测、专家内部试用、50–100 名学术客户邀请试点、最高不超过 10% 真实流量。任一安全门禁退化时停止放量或回滚。
- 可信 MVP 的建议技术分层包括：三语客户端与专家工作台、API 与身份层、科研咨询编排与项目状态层、风险门控与 Reviewer、混合检索与知识治理、关系型数据库与缓存、对象存储、可观测性和评测平台。
- 结构化业务对象必须通过版本化 schema 校验；关键建议证据覆盖率为 100%；跨语言实体、数值、单位和风险等级在切换语言后保持一致。
- 性能目标为无外部工具的首次流式响应 P95 小于 3 秒、检索型回答首个有效进度 P95 小于 5 秒；所有长任务支持检查点、取消、超时、重试和幂等。
- 12 周实施按门禁推进：第 1–2 周完成数据盘点、术语、风险矩阵和金标；第 3–4 周完成项目事实与混合检索；第 5–6 周完成决策卡、Reviewer、风险分级和三语；第 7–8 周完成专家工作台、转接和质量事件；第 9–10 周完成 CRM 授权与运营评测；第 11 周回归、安全与灰度演练；第 12 周完成邀请试点和上线评审。

## Testing Decisions

- 采用一个最高层测试接缝：一次完整科研咨询旅程。测试从客户创建或恢复科研项目开始，经项目事实确认、最小必要追问、跨语言循证检索、科研 Reviewer、风险判定、科研决策卡发布或强制转接、专家处理、PI 审批、授权式转化和反馈结束。
- 好测试只断言外部可观察行为、持久化业务状态、事件和集成契约，不断言提示词文本、内部链路步骤、模型思维过程、具体类名或私有函数调用。
- 主验收对象是科研项目状态、科研决策卡、证据绑定、风险等级、专家工单、授权事件和质量事件。聊天措辞只在涉及风险提示、限制、引用和禁止性 CTA 时进行必要断言。
- 端到端旅程使用可控的模型、检索、知识、CRM 和工单替身，固定输入与证据集合；真实模型表现通过 NovaBench 离线评测和影子评测验证，避免将概率模型结果混入确定性业务验收。
- UAT-01 验证中文 FFPE/RNA 标准咨询能够完成项目建档、最小必要追问、主备方案、平台选择和正式决策卡。
- UAT-02 验证同一项目从英文开始、切换日文继续时，项目事实、数值、风险与证据标识保持一致。
- UAT-03 验证缺失 DV200 等关键条件时只输出条件性路径，并明确检测方法与方案分支，不伪造确定参数。
- UAT-04 验证内部 SOP 与外部文献发生实质冲突时展示冲突、停止最终建议并创建一次性交接包。
- UAT-05 验证极低质量或非标准样本触发强制转接，不出现可直接执行的 CTA。
- UAT-06 验证专家认领、退回补充、修改、批准和发布新版本的完整行为与 SLA 事件。
- UAT-07 验证低风险标准案例在证据门禁通过后由 AI 发布正式决策卡。
- UAT-08 验证 DOI、PMID、文献标题、数值单位或证据适用性错误会被 Reviewer 阻断并要求补证。
- UAT-09 验证未授权咨询不会创建 CRM 客户或商机，也不会向销售暴露客户材料。
- UAT-10 验证客户主动获取报价、预约专家或同意联系后创建授权事件，并以幂等方式写入 CRM。
- UAT-11 验证专家修改与失败咨询只生成候选知识，不会自动进入正式知识或改变生产回答。
- UAT-12 验证回答低分、客户纠错、引用失效或重复开单生成质量事件，并可被认领、归因和关闭。
- UAT-13 验证外部模型调用只包含策略允许的脱敏问题和公开内容；敏感科研数据被数据出域策略阻断。
- UAT-14 验证工单或 CRM 暂时不可用时进入补偿队列，恢复后幂等补写，不产生重复工单或商机。
- UAT-15 验证证据失效后关联科研决策卡进入待复核状态并通知知识 Owner，不静默保留正式状态。
- NovaBench 金标集在可信 MVP 前冻结 300–600 条，覆盖标准 FFPE/RNA、条件缺失、证据冲突、极低质量和非标样本、内部与外部冲突、中英日改写与混合语言、应转人工与可自动解决、科学最优但不等于诺禾主推服务等案例。
- 检索层测试评估召回、重排、权限过滤、版本有效性和 DOI/PMID 可解析性；回答层评估结论正确性、证据覆盖、适用边界、数值单位和引用支持关系。
- 风险层测试评估高风险转接召回率、强制转接后禁止性输出、风险等级一致性和低风险误转接；业务层测试评估安全自助解决率、专家 SLA、重复开单、CSAT 和授权式转化。
- 跨语言测试使用同一科研项目的中、英、日等价表达，断言统一实体、项目事实、证据 ID、风险等级和决策卡状态一致，同时允许自然语言表达差异。
- 权限与合规测试覆盖跨租户访问、越权审批、专家未认领访问、销售读取未授权数据、撤回授权、数据删除、恶意文件、日志泄露和外部模型出域。
- 集成契约测试覆盖 CRM、工单、知识平台、PubMed/Crossref、对象存储和身份系统，重点验证幂等、事件乱序、超时、重试、补偿和状态对账。
- 性能测试覆盖首个流式响应、检索进度、并发咨询、长任务取消和专家工作台队列；故障演练覆盖模型、检索、数据库、工单、CRM 和知识平台不可用。
- 发布验收按历史影子、专家内部、邀请客户和最高 10% 灰度逐级执行。引用有效率低于 98%、高风险转接召回率低于 95%、自信错答恶化、出现 P0 阻断缺陷或敏感数据事件时，发布失败并回滚。
- 当前项目没有既有应用代码或测试框架，因此不存在可复用的代码级测试先例；本规格复用 PRD 已确认的 15 条 UAT、NovaBench 金标评测和门禁式灰度作为最高层先例。实现阶段应优先构建一个可编排的咨询旅程测试夹具，再为纯 schema、权限策略和集成适配器补充必要的低层契约测试。

## Out of Scope

- 可信 MVP 不执行完整生物信息分析流水线，只提供分析方法推荐与后续路线；Nextflow、Snakemake 等可复现执行属于后续阶段。
- 不控制实验室设备，不自动修改样本处理、实验参数或湿实验操作。
- 不提供临床诊断、治疗建议、面向患者的医疗决策或人类遗传资源合规结论。
- 不自动报价、签约、承诺交付周期或创建未授权 CRM 商机。
- 不在可信 MVP 开放 A3 高自主多步规划进入真实客户链路；A3 仅可在沙箱评测优于 A2 且可安全退化后考虑。
- 不允许在线训练或自动更新模型权重，不允许智能体自行修改代码、生产工作流或正式 SOP。
- 不在首期提供实时语音、视频、企业批处理、企业组织级 SLA 与完整多租户管理体验。
- 不在首期建立 Neo4j 等独立关系图数据库；统一科研实体和关系可先由关系型数据库与检索索引承载。
- 不把项目事实、科研决策卡、CRM 客户、专家工单和正式 SOP 合并到一个主数据系统。
- 不以减少转人工、对话数量、页面访问或模型置信度作为成功指标。
- 不把服务适配说明伪装为科学结论，也不保证诺禾产品永远是科学最优方案。
- 不承诺试点业务目标在缺少真实基线时必然达成；安全发布门禁仍是进入和扩大灰度的必要条件。

## Further Notes

- 本规格继承已经确认的领域词汇与 12 项架构决策；实施不得用“聊天摘要”替代科研决策卡，不得用“自助率”替代安全自助解决率，也不得用“自动派单”替代专家认领。
- 当前目录主要是竞赛方案和产品规格，不是已有应用代码仓库。工程智能体开始实现前，应先建立最小可运行骨架、领域 schema、事件契约和一次完整科研咨询旅程测试夹具。
- 推荐的首个垂直切片是：中文 FFPE/RNA 标准咨询 → 项目事实确认 → 三个以内最小必要追问 → 循证检索 → 低风险正式科研决策卡 → PI 审批。第二个切片应立即覆盖证据冲突 → 强制转接 → 专家批准，避免先构建只有顺利路径的演示。
- 任何实现若需要改变系统主责边界、强制转接语义、风险分级审批、三语统一知识、数据出域规则或自进化治理，必须先新增或重开 ADR。
- `ready-for-agent` 表示该规格已具备拆分实施任务的条件，不表示安全门禁、内部 SOP、真实 CRM／工单接口或专家排班已经到位；这些仍是 12 周计划中的外部依赖。
